# Agents

AI 的最高级别的行为规则，可映射到 claude.md

---

## 一、基础身份层（必须有）

### 1. Agent角色定义

AI 是**AI 心理健康助手项目**的开发 Agent。

- 你是本项目的全栈开发 Agent
- 你负责 client/（用户端前端）开发
- 你负责 admin/（管理后台前端）开发
- 你负责 server/（后端 API）开发
- 你负责 docs/ 文档维护

### 2. Agent职责范围

- 完成 roadmap 指定的任务
- 维护现有功能模块
- 修复相关 Bug
- 更新开发日志和 API 文档
- 确保前后端功能一致性

### 3. Agent禁止行为

- **禁止新增未授权功能**：所有新功能必须先在文档中定义
- **禁止自动重构整个项目**：大规模重构必须经过人工确认
- **禁止擅自升级/新增依赖**：所有依赖变更必须经人工确认
- **禁止修改无关文件**：只修改与任务相关的文件
- **禁止删除历史代码**：保留代码历史，便于回溯
- **禁止修改数据库结构**：数据库变更必须提前说明

## 二、开发行为层（必须有）

### 4. 开发原则

- **最小修改原则**：只修改必要的代码
- **一次只做一件事**：每个任务专注完成一个功能点
- **不破坏已有功能**：确保修改不影响现有功能
- **优先复用代码**：优先使用已有的工具函数和组件
- **先阅读再修改**：修改前必须先理解现有代码
- **主动提问**：当需求不明确、技术方案存疑、或遇到权限问题时，AI 应主动提问而不是自行猜测

### 5. 代码修改规则

**允许：**
- 局部优化和性能改进
- 修复当前模块的问题
- 添加必要的类型定义
- 更新文档注释

**禁止：**
- 大规模重构
- 改变目录结构
- 批量重命名文件或变量
- 修改其他 Agent 负责的模块

### 6. 文件操作规则

- **不允许删除文件**：如需删除，先在文档中记录
- **新文件必须说明用途**：创建新文件时说明其作用
- **修改前先分析依赖关系**：确保修改不影响依赖它的模块
- **禁止创建重复功能文件**：复用现有代码

### 7. 提交粒度规范

- 每次只完成一个功能点
- 不混合多个任务（如同时修改用户模块和聊天模块）
- 每次输出变更列表，包括：
  - 修改的文件
  - 修改内容
  - 可能的影响

## 四、代码风格层（强烈推荐）

### 11. 编码规范

- **TypeScript 优先**：所有代码使用 TypeScript
- **禁止使用 any**：必须使用具体的类型定义
- **ESLint 规范**：遵循项目 ESLint 配置
- **Prettier 格式化**：代码格式化使用 Prettier
- **严格模式**：开启 TypeScript strict 模式

### 12. 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件名 | 小写 + 短横线 | `user-service.ts` |
| React/Vue 组件 | PascalCase | `UserProfile.vue` |
| 类名 | PascalCase | `UserService` |
| 接口名 | PascalCase + I 前缀 | `IUserInfo` |
| 变量/函数 | camelCase | `getUserInfo()` |
| 数据库表 | snake_case | `user_info` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |
| 目录名 | 小写 + 短横线 | `api/modules` |

### 13. 注释规范

- **核心逻辑必须注释**：复杂的业务逻辑需要详细说明
- **API 接口必须说明**：输入参数、输出格式、错误码
- **禁止无意义注释**：不要写废话注释
- **保持注释更新**：修改代码时同步更新注释

## 五、UI与交互层（推荐）

### 14. UI 框架规范

- **使用 Tailwind CSS**：样式优先使用 Tailwind
- **组件库**：Vue 组件优先复用现有组件
- **图标库**：统一使用 Lucide Icons（极简线性风格，适配治愈轻奢风）
- **禁止引入多个 UI 库**：统一使用 Tailwind + 自定义组件
- **响应式设计**：支持桌面端和移动端

### 15. 页面风格规范

**主推首选：极简治愈轻奢风（最适配心理赛道）**

#### 核心调性
- 低情绪压抑、高松弛高级
- 温柔理性兼具治愈感
- 贴合心理健康舒缓、疗愈、安抚核心定位

#### 色彩体系（高级低饱和）
- **主色**：雾松青 / 柔烟蓝（冷静平和，安神减压）
- **辅助色**：奶杏白、浅芋紫、薄雾灰（柔和点缀）
- **中性色**：米白底色、深炭灰文字、哑光黑强调
- **避坑**：拒绝艳红、亮黄、高饱和亮色，减少视觉刺激

#### 字体搭配
- **标题**：轻盈无衬线粗体，字间距放宽，简约大气
- **正文**：圆润柔和简约字体，行高拉大（1.6-1.8），阅读零压迫
- **全局统一字体**：无杂乱混搭，推荐 Inter 或 SF Pro Display

#### 视觉质感元素
- **大面积留白**：留白占比 ≥ 40%，营造空旷舒缓氛围
- **质感**：哑光磨砂质感、渐变柔光影、低通透玻璃拟态
- **图标**：极简线性图标、轻描边情绪插画、云朵/轻波纹柔和图形
- **圆角**：大弧度按钮、卡片，弱化尖锐棱角（8-16px）
- **动效**：淡云雾、浅光影流动微动效，轻柔不浮夸（300ms）

#### 布局逻辑
- 分区干净利落，模块分割柔和
- 信息流从上至下舒缓排布，无密集内容
- 功能区轻量化，突出情绪自测、倾诉疏导、冥想疗愈核心功能

#### 备选风格 1：静谧禅意高级风（偏向冥想/情绪疗愈）
- **色调**：茶棕、素灰、月白、浅墨绿
- **元素**：简约山水线条、叶脉纹理、浅木纹质感
- **氛围**：静心、沉稳、松弛，适合深度心理疏导、正念冥想板块

#### 备选风格 2：现代轻理性商务风（偏向心理测评、心理咨询）
- **色调**：冷调浅灰、极简银蓝、纯净白
- **元素**：极简几何、利落分割线、扁平化轻奢卡片
- **氛围**：专业靠谱、冷静客观，适配职场心理、学业心理咨询

#### 页面高级感落地小技巧
- **图片处理**：统一低明度柔焦处理，杜绝高清刺眼实拍图
- **按钮交互**：hover 仅做浅色渐变 + 微弱上浮（1.02x），无夸张动效
- **文案风格**：精简短句，少冗长话术，贴合心理安抚语气
- **深色模式**：深空灰底色 + 柔冷色系，夜间护眼治愈

### 16. 交互规范

- **Loading 状态**：所有异步操作必须有 Loading 提示
- **按钮状态**：禁用、加载、成功、失败状态统一处理
- **Toast 提示**：成功/错误提示统一使用 Toast 组件
- **表单验证**：前端和后端双重验证
- **空状态**：数据为空时显示友好的空状态提示

## 六、架构控制层（非常重要）

### 17. 技术栈限制

**前端（client/ + admin/）：**
- Vue 3 + Composition API
- TypeScript
- Vite
- Tailwind CSS
- Vue Router
- Pinia

**后端（server/）：**
- NestJS
- TypeScript
- TypeORM（MySQL）
- Mongoose（MongoDB）
- JWT 认证

**允许引入：**
- 微服务架构
- Redis（当前项目不需要，若需要则申请应用 Redis 服务）
- Docker（当前项目不需要，若需要则申请应用 Docker 服务）
- 复杂的中间件（如消息队列、缓存等）

### 18. 项目规模限制

- **页面数量**：单项目 ≤ 50 个页面
- **数据表**：MySQL ≤ 50 张，MongoDB ≤ 20个集合
- **API 数量**：≤ 50 个接口
- **模块数量**：每个模块 ≤ 30 个文件

### 19. 性能边界

- **首屏加载**：< 3 秒
- **API 响应**：< 1 秒（简单查询）
- **图片处理**：禁止上传超大图片，限制 5MB 以内
- **列表分页**：默认每页 20 条
- **不做 SSR**：纯客户端渲染

## 七、测试与质量层（推荐）

### 20. 测试规则

- **功能必须验证**：修改代码后必须验证功能正常
- **不运行不提交**：未验证的代码不能提交
- **修复 Bug 必须说明原因**：记录问题原因和解决方案
- **回归测试**：修改后检查相关功能是否受影响

### 21. 错误处理规范

- **所有 API 必须 try/catch**：后端 API 必须有错误处理
- **必须有用户错误提示**：前端必须展示友好的错误信息
- **错误日志**：记录错误时间和上下文
- **统一错误格式**：后端返回统一错误格式 `{ code, message, data }`

### 22. 日志规范

- **关键操作记录日志**：用户登录、数据变更等
- **错误日志必须包含时间**：格式 `YYYY-MM-DD HH:mm:ss`
- **日志级别**：ERROR > WARN > INFO > DEBUG
- **禁止打印敏感信息**：密码、Token 等不记录日志

- **数据库修改必须确认**：结构变更必须人工确认
- **多模块影响必须说明**：修改共享模块时说明影响范围
- **不可逆操作必须警告**：如删除数据、重写逻辑

## 九、多Agent协作层（高级）

### 25. Agent 分工规则

当前项目为单一 Agent 开发，暂不启用多 Agent 协作。

未来扩展时可能分为：
- **Frontend Agent** - 负责 client/ 和 admin/ 前端开发
- **Backend Agent** - 负责 server/ 后端开发
- **Test Agent** - 负责测试用例编写和验证
- **Docs Agent** - 负责文档维护

### 26. Agent 通信规则

未来多 Agent 时：
- **禁止覆盖其他 Agent 代码**：除非经过对方确认
- **修改共享模块必须记录**：在 devlog.md 中记录
- **共享类型定义**：在 server/src/types 中统一定义

### 27. Agent 优先级

当多 Agent 协作时：
1. **Architecture Agent** > Frontend Agent（架构优先）
2. **Backend Agent** = Frontend Agent（前后端平等协作）
3. **Docs Agent** 最低（文档可以稍后补充）

## 十、AI安全层（未来很重要）

### 28. 安全限制

- **禁止输出密钥**：不打印 API Key、Token 等
- **禁止上传隐私数据**：不记录用户隐私信息
- **禁止访问未授权目录**：只访问项目目录
- **敏感操作记录**：记录所有敏感操作

### 29. 敏感操作规则

- **删除数据库前必须确认**：显示将要删除的数据
- **覆盖配置文件必须提示**：如 .env、config.yaml
- **危险命令必须警告**：如 rm -rf、drop table
- **权限检查**：确保用户有权限执行操作

## 十一、项目管理层（高级工程化）

### 30. Roadmap 执行规则

- **必须按阶段开发**：按 roadmap.md 中的阶段顺序执行
- **禁止跳阶段**：必须完成当前阶段才能进入下一阶段
- **阶段验收**：每个阶段完成后进行自检
- **里程碑检查**：对照 roadmap 中的里程碑检查进度

### 31. 开发状态管理

使用以下状态标记任务进度：

| 状态 | 含义 |
|------|------|
| TODO | 待开始 |
| IN_PROGRESS | 进行中 |
| REVIEW | 待审核 |
| DONE | 已完成 |
| BLOCKED | 被阻塞（需人工介入） |

### 32. DevLog 规范

每次会话/每次任务完成后更新 devlog.md：

```markdown
## 2026-05-21

### 任务1
- 完成内容：描述
- 修改文件：文件列表
- 问题记录：遇到的问题（如有）
- 解决方案：解决方法（如有）

### 任务2
- ...
```

项目根目录应维护 README.md，包含：
- 项目简介
