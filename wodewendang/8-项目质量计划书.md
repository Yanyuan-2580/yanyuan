## 一、引言

### 1.1 编制目的

本质量计划书为「心理健康AI助手」项目制定系统化的质量管理框架，明确质量目标、标准、检查清单和改进机制，确保项目在功能完整性、运行稳定性、界面美观性、文档齐全性和安全可靠性五个维度达到预期质量水平。

### 1.2 适用范围

本计划适用于项目的以下组成部分：

| 组成部分 | 目录 | 技术栈 |
|----------|------|--------|
| 用户端前端 | `client/src/` | Vue 3 + Vite + Pinia + Tailwind CSS + Vue Router + Lucide Icons |
| 管理后台前端 | `admin/src/` | Vue 3 + Vite + Element Plus + Pinia + Tailwind CSS |
| 后端服务 | `server/src/` | NestJS + TypeORM (MySQL) + Mongoose (MongoDB) + Redis + JWT |
| 项目文档 | `docs/` | Markdown 格式，中文编写 |
| AI 集成 | `server/src/shared/ai/` | DeepSeek API / OpenAI 兼容接口 |
| 风控系统 | `server/src/shared/risk-control/` | 高危词库 + 中危词库 + 正则匹配 + 风险升级追踪 |

### 1.3 项目背景

本项目是一个全栈心理健康AI助手应用，整合 AI 对话陪伴、情绪追踪、知识库、冥想放松、日记反思、心理测评、视频咨询和管理后台等功能，为用户提供一站式的在线心理健康服务。

### 1.4 当前开发阶段

- **已完成**: Phase 1-4（v1.0 ~ v2.7.0），包含 12 个业务模块、26 个用户端页面、5 个管理端页面、50+ 个 API 端点。
- **进行中**: v2.7.0 全局 UI 美化（已完成）。
- **待开发**: Phase 5（微信小程序）、Phase 6（提醒系统 + RBAC + 文件上传）。

## 二、质量目标

### 2.1 总体质量方针

本项目遵循"功能完整、运行稳定、界面美观、文档齐全、安全可靠"五位一体的质量方针，以用户心理健康服务的特殊场景为出发点，将质量要求贯穿于需求分析、设计开发、测试验证、交付上线的全流程。

### 2.2 量化质量目标

| 维度 | 目标 | 量化指标 | 测量方式 |
|------|------|----------|----------|
| **功能完整性** | 所有需求功能上线可用 | 需求覆盖率 ≥ 95% | 对照 requirements.md 逐项核验 |
| **运行稳定性** | 核心功能持续稳定运行 | API 可用率 ≥ 99.5%，首屏加载 ≤ 3s，API 响应 ≤ 1s | 性能监控 + 错误日志统计 |
| **界面美观性** | 统一遵循设计规范 | 26 个页面视觉统一率 100%，设计走查通过率 ≥ 95% | 设计走查清单核验 |
| **文档齐全性** | 关键文档完整且与代码同步 | 6 类核心文档齐全，API 文档与实现一致率 100% | 文档审查清单核验 |
| **安全可靠性** | 用户数据安全，风控有效 | 高危内容拦截率 ≥ 95%，敏感数据加密率 100%，无 P0 安全漏洞 | 安全测试 + 风控命中统计 |

### 2.3 各模块质量目标

| 模块 | 功能目标 | 性能目标 | 安全目标 |
|------|----------|----------|----------|
| **User** | 注册/登录/密码/资料功能正常 | 登录响应 < 500ms | 密码 bcrypt 加密，Token 安全存储 |
| **Chat** | 流式对话 + 多会话管理正常 | SSE 首字延迟 < 2s | 风控检测覆盖 100% 消息 |

| **Mood** | 记录/历史/统计正常 | 统计查询 < 500ms | 用户数据权限隔离 |
| **Meditation** | 列表/详情/记录/统计正常 | 列表加载 < 300ms | 无敏感数据暴露 |

## 三、质量保证组织

### 3.1 质量角色与职责

| 角色 | 人员 | 职责 |
|------|------|------|
| **项目负责人** | Developer | 制定质量方针和目标，审批质量计划，对最终质量负责 |
| **开发 Agent** | AI Agent | 遵循开发规范编码，执行自测和单元验证，修复质量问题 |
| **质量审查人** | Developer（人工） | 审查关键代码变更，执行设计走查，确认质量标准达标 |
| **文档维护人** | AI Agent + Developer | 开发后同步更新文档，定期审查文档准确性 |

### 3.2 当前团队模式

本项目当前为 **单人开发 + AI Agent 辅助** 模式。质量保证的重点在于：

- AI Agent 严格遵循 `CLAUDE.md` 和 `docs/Agents.md` 中的开发规范。
- 每一轮修改后由 Developer 进行人工代码审查（Code Review）。
- 关键变更（数据库结构、认证逻辑、部署配置）必须经过人工确认。

### 3.3 沟通机制

| 环节 | 方式 | 频率 |
|------|------|------|
| 任务汇报 | AI Agent 输出任务完成报告（含修改文件、风险评估） | 每次任务完成后 |
| 质量检查 | Developer 对照质量检查清单核验 | 每个版本发布前 |
| 文档审查 | Developer 检查文档与代码一致性 | 每 2 个版本 |
| 问题记录 | 记录到 `devlog.md` | 发现问题时即时记录 |

## 四、质量标准

### 4.1 功能质量标准

| 编号 | 标准项 | 具体要求 |
|------|--------|----------|
| F-01 | 需求覆盖 | 所有 `docs/requirements.md` 中标记为 P0/P1 的需求均已实现 |
| F-02 | API 正确性 | API 端点返回的 `code`、`message`、`data` 格式符合规范，业务逻辑正确 |
| F-03 | 前后端一致 | 前端调用的 API 参数和后端期望的参数一致，无字段名不匹配 |
| F-04 | 边界处理 | 空数据、超长输入、特殊字符等边界情况有正确处理 |
| F-05 | 错误处理 | 所有 API 有 try/catch，前端有用户友好的错误提示 |
| F-06 | 权限控制 | 未登录用户无法访问需认证的页面和 API，用户间数据隔离 |
| F-07 | 状态管理 | 前端 Pinia store 的状态更新与后端数据一致，无脏数据残留 |
| F-08 | 分页正确 | 列表分页的 total、page、pageSize、totalPages 计算准确 |
| F-09 | 搜索功能 | 搜索关键词能正确匹配并返回结果，空关键词返回全部数据 |
| F-10 | 文件命名 | 文件名、组件名、函数名遵循 kebab-case / PascalCase / camelCase 规范 |
| F-11 | 类型定义 | TypeScript 类型定义完整，无 `any` 类型残留 |

### 4.2 文档质量标准

| 编号 | 标准项 | 具体要求 |
|------|--------|----------|
| D-01 | CLAUDE.md | 项目概述、技术栈、目录结构、当前进度、开发约定均准确且与代码一致 |
| D-02 | requirements.md | 功能需求完整描述，优先级标记明确 |
| D-03 | api.md | 所有 API 端点的路径、方法、参数、响应格式与实际代码一致 |
| D-04 | devlog.md | 每次开发任务的日期、内容、文件、问题记录完整 |

### 4.3 UI 质量标准

| 编号 | 标准项 | 具体要求 |
|------|--------|----------|
| U-01 | 色彩统一 | 所有页面使用 Tailwind 配置中的 calm/emerald 色系，无硬编码颜色 |
| U-02 | 圆角统一 | 按钮 12px、卡片 16px、输入框 10px、弹窗 20px |
| U-03 | 图标统一 | 全局使用 Lucide Icons，无混用其他图标库 |
| U-04 | 响应式 | 桌面端（≥1024px）和移动端（375-768px）均正常显示，无布局错乱 |
| U-05 | 交互反馈 | Loading 状态、空状态、错误状态、成功状态均有对应的 UI 反馈 |

### 4.4 性能质量标准

| 编号 | 标准项 | 量化指标 |
|------|--------|----------|
| P-01 | 首屏加载 | < 3 秒（Lighthouse Performance Score ≥ 80） |
| P-02 | API 响应 | 简单查询 < 1 秒，复杂统计 < 3 秒 |
| P-03 | SSE 流式 | AI 回复首字延迟 < 2 秒 |
| P-04 | 图片加载 | 图片压缩后 < 500KB，使用懒加载 |
| P-05 | 并发支持 | 支持 ≥ 1000 并发用户（理论值） |

### 4.5 安全质量标准

| 编号 | 标准项 | 具体要求 |
|------|--------|----------|
| S-01 | 密码安全 | 使用 bcrypt 哈希存储，不可逆 |
| S-02 | Token 安全 | accessToken 2h 过期，refreshToken 7d 过期，HTTPS 传输 |
| S-03 | 输入校验 | 前后端双重校验，防止 SQL 注入 / XSS 攻击 |
| S-04 | 风控覆盖 | AI 对话、日记、评论 3 个入口 100% 覆盖风控检测 |
| S-05 | 数据隔离 | 用户只能访问自己的数据，管理员操作需日志记录 |
| S-06 | 敏感信息保护 | 日志中不打印密码、Token、手机号等敏感信息 |

## 七、质量改进机制

### 7.1 问题发现与记录

| 来源 | 记录方式 | 责任人 |
|------|----------|--------|
| 开发过程中的技术问题 | 记录到 `devlog.md`，标注"问题记录-解决方案" | AI Agent |
| 测试发现的 Bug | 记录到 `devlog.md`，标注 Bug 描述/根因/修复方案 | Developer |
| 用户反馈的问题 | 记录到 `devlog.md`，标注反馈来源和优先级 | Developer |
| 代码审查发现的问题 | 记录到审查报告，附修复建议 | Developer |

### 7.2 问题分级与处理时效

| 等级 | 定义 | 示例 | 处理时效 |
|------|------|------|----------|
| **P0-紧急** | 核心功能不可用，数据安全问题 | 登录失败、数据泄露、服务宕机 | 4 小时内修复 |
| **P1-高** | 重要功能异常，用户体验严重受影响 | AI 对话无法回复、日记保存失败 | 24 小时内修复 |
| **P2-中** | 一般功能缺陷，有替代方案 | UI 显示错位、非关键数据统计错误 | 3 个工作日内修复 |
| **P3-低** | 轻微问题，不影响使用 | 文案错别字、颜色偏差、建议性优化 | 下一版本修复 |

### 7.3 版本质量回顾

每个版本发布后，进行以下回顾活动：

1. **版本质量总结**：
   - 本版本修复了多少问题（按 P0-P3 分类统计）。
   - 本版本引入了多少新问题（如果有）。
   - 质量目标达成情况（对照第二节量化指标）。

2. **质量趋势分析**：
   - 对比上一版本，质量指标是上升还是下降。
   - 分析趋势变化的原因（如：新增功能较多导致 Bug 增加、重构改善了稳定性等）。

3. **改进措施制定**：
   - 针对反复出现的问题类型，制定预防措施（如：补充自动化检查、强化代码审查重点）。
   - 针对新发现的质量风险，更新检查清单。

### 7.4 持续改进循环

采用 PDCA 循环推动质量持续改进：

```
Plan（计划）→ Do（执行）→ Check（检查）→ Act（改进）
     ↑                                          |
     └──────────────────────────────────────────┘
```

| 阶段 | 活动 |
|------|------|
| **Plan** | 制定本版本质量目标，更新质量检查清单 |
| **Do** | 按质量计划执行开发、测试、审查活动 |
| **Check** | 版本发布前核验检查清单，统计质量指标，进行版本回顾 |
| **Act** | 根据回顾结果调整质量计划，更新标准和流程，进入下一轮 PDCA |

### 7.5 质量档案

每个版本维护以下质量档案：

| 档案 | 内容 | 存放位置 |
|------|------|----------|
| 版本检查清单结果 | 功能/文档/UI/安全 检查清单的逐项结果 | `devlog.md` |
| 问题清单 | 本版本发现的所有问题及处理状态 | `devlog.md` |
| 质量指标数据 | API 可用率、首屏加载时间、风控拦截率等 | `devlog.md` |
| 改进措施跟踪 | 上版本改进措施的执行情况和效果 | `devlog.md` |

### 7.6 质量培训与规范宣贯

- 每次更新 `CLAUDE.md` 或 `docs/Agents.md` 后，AI Agent 必须在下一轮对话开始时重新读取更新后的规范文件。
- 新增开发规范或质量标准时，在 `devlog.md` 中记录变更并说明原因。
- 每 5 个版本进行一次全面的规范审查，检查是否有过时或矛盾的规则需要更新。
