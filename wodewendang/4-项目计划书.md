## 一、总体计划

### 1.1 项目概述

"心理健康AI助手"项目旨在构建一个基于人工智能技术的综合性数字心理健康服务平台。项目采用敏捷迭代开发方法，规划为6个阶段（Phase 1—6），总工期12个月（2026年1月—2026年12月）。当前已进入Phase 5阶段，已完成Phase 1—4的全部开发任务，系统版本为v2.7.0。

### 1.2 项目目标

**总体目标**：在2026年12月前完成从Web端到微信小程序的全平台覆盖，交付一个安全、稳定、用户体验优秀的AI心理健康综合服务平台，实现从项目开发到正式运营的完整闭环。

**阶段性目标**：

| 阶段 | 目标描述 | 关键交付物 | 完成标准 |
|------|---------|-----------|---------|
| Phase 1 (已完成) | 搭建项目骨架 | 项目框架 + 数据库设计 + 用户认证 | 用户可注册登录、Token机制运转正常 |
| Phase 2 (已完成) | 核心功能上线 | AI对话 + 日记 + 心情 + 冥想 + 知识库 | 六大核心模块功能完整可用 |
| Phase 3 (已完成) | 扩展功能上线 | 测评 + 视频 + 通知 + 风控 + 管理后台 | 全功能模块完成集成测试 |
| Phase 4 (已完成) | 优化完善 | 报告导出 + 评论 + 搜索 + UI打磨 + 性能优化 | 通过系统测试和性能基准 |
| Phase 5 (当前) | 微信小程序 | Uni-App项目 + 核心页面适配 + 微信登录 | 小程序通过审核上线 |
| Phase 6 (规划中) | 最终打磨 | 提醒系统 + RBAC + 审计日志 + 文件上传 | v3.0正式发布 |

### 1.3 项目制约因素

- **时间制约**：总工期12个月，2026年12月必须完成全平台发布
- **资源制约**：核心团队7—10人，总预算80—120万元
- **技术制约**：AI能力依赖第三方API（DeepSeek），需要配置备用方案
- **合规制约**：必须遵守《个人信息保护法》《数据安全法》等法律法规
- **质量制约**：心理健康赛道的特殊性要求AI安全性和稳定性标准高于一般互联网产品

## 二、项目范围说明书

### 2.1 范围基准

**项目包含范围**：
- 用户端Web应用（22个页面，响应式设计）
- 管理后台Web应用（5个核心页面，可扩展至10+页面）
- NestJS服务端（11个业务模块，14个TypeORM实体，MySQL + MongoDB + Redis）
- AI引擎集成（DeepSeek API接入，SSE流式 + 非流式回退）
- 风控系统（关键词匹配 + 风险升级追踪 + 自动干预）
- 通知系统（站内通知 + SMS/Email接口预留）
- 微信小程序（Uni-App框架，核心页面适配）
- 技术文档（架构文档、API文档、数据库文档、部署文档）
- 用户协议和隐私政策

**项目不包含范围**：
- 专业心理咨询师真人服务（平台只提供工具，不雇佣咨询师）
- 医疗诊断功能（产品定位为心理健康陪伴和自助工具，不涉及疾病诊断）
- iOS/Android原生App（当前仅Web + 小程序，原生App不在本阶段范围）
- 第三方内容授权购买（知识库内容为原创或使用公开领域资料）
- 海外多语言版本（当前仅限中文，国际化在后续项目规划中）
- 硬件设备集成（如VR头显、生物反馈设备等不在本期范围）
- 保险理赔对接（心理健康相关保险理赔数字化不在本期范围）

### 2.2 范围管理

**范围变更控制流程**：
1. 任何范围变更请求需提交书面变更申请（Change Request）
2. 项目经理评估变更对进度、成本、质量的影响
3. 变更影响评估报告提交项目决策层审批
4. 获批准的变更更新至需求跟踪矩阵和项目计划
5. 未获批准的变更记录至变更日志并归档
6. 紧急变更（如安全漏洞修复）可经项目经理快速审批后实施，事后补全流程

**范围验证方式**：
- 每阶段结束时进行阶段成果评审（Phase Review）
- 核心功能完成后进行用户验收测试（UAT）
- 所有功能需求追溯至需求规格说明书中的需求条目
- 使用需求跟踪矩阵（RTM）确保每条需求有对应的设计、实现和测试

### 4.2 成本控制措施

1. **分阶段投入**：每阶段结束后进行成本核算和预算调整，避免一次性投入过多
2. **云服务弹性伸缩**：开发测试阶段使用低配置实例，正式上线后再扩容
3. **开源工具优先**：优先使用开源免费工具（Vite/NestJS/Vue/Tailwind/Lucide Icons等），避免商业授权费用
4. **AI API用量优化**：通过缓存常见回复、限制免费用户每日对话次数等方式控制API成本
5. **远程协作**：优先远程办公，减少办公场地租赁成本
6. **成本预警**：设立月度成本预警线（月度实际成本 > 预算120% 时触发审查）

### 9.2 版本管理策略

**语义化版本规范（SemVer 2.0）**：
- 版本格式：`v<MAJOR>.<MINOR>.<PATCH>`
- MAJOR（主版本）：不兼容的API修改、重大架构变更
- MINOR（次版本）：向下兼容的功能新增
- PATCH（修订版本）：向下兼容的问题修复

**分支管理策略（GitFlow简化版）**：
- `main` — 生产就绪代码，只接受来自 `develop` 的合并，每次合并打Tag
- `develop` — 开发主线
- `feature/{功能名}` — 功能开发分支，从 `develop` 分出，完成后合并回 `develop`
- `hotfix/{问题描述}` — 紧急修复分支，从 `main` 分出，修复后合并回 `main` 和 `develop`
- `release/v{版本号}` — 发布准备分支，从 `develop` 分出，测试通过后合并到 `main`

### 9.3 变更管理流程

1. **变更请求发起**：任何团队成员或干系人可发起变更请求，提交至项目经理
2. **变更影响分析**：项目经理组织相关人员评估变更对范围、进度、成本、质量、风险的影响，形成变更影响报告（24—48小时内完成）
3. **变更决策**：
   - 影响范围小（< 2人天工作量）且不改变里程碑：项目经理可直接审批
   - 影响范围中等（2—10人天）：需技术负责人+项目经理共同审批
   - 影响范围大（> 10人天或改变里程碑）：需提交决策委员会审批
4. **变更实施**：获批准的变更纳入当前或下个迭代计划，更新需求跟踪矩阵和项目计划
5. **变更验证**：变更完成后通过测试验证，确认达到预期效果
6. **变更记录**：所有变更（无论批准与否）均记录在变更日志中，包含：变更编号、日期、描述、影响分析、决策结果、决策人

### 9.4 配置审计

- **代码审计**：每个PR合并前需通过代码审查、Lint检查和自动化测试
- **依赖审计**：每月运行 `npm audit` 检查依赖安全漏洞，高危漏洞3个工作日内修复
- **配置审计**：每阶段结束时检查所有配置项是否与文档一致
- **发布审计**：每次正式发布前确认所有配置项（环境变量、数据库连接、API密钥等）正确无误
- **基线建立**：每个里程碑交付后建立配置基线，作为后续变更的参考基准
